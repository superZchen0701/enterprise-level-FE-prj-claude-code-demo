# OpenSpec 使用指南

## 一、环境准备

### node版本

- Node.js 20.19.0 or higher。
- 可以使用 nvm 来管理 node 版本。

### 全局安装 OpenSpec

```shell
npm install -g @fission-ai/openspec@1.2.0
openspec --version
输出版本号说明安装成功
```

## 二、各指令使用方式+实战场景案例

### /opsx:explore - 探索模式

**使用方式**：

- 核心作用：仅用于需求讨论，处于“只读模式”，不生成任何文件、不创建任务，也不执行开发操作。
- 使用前提：需求不明确（如产品未确定功能细节）、需求复杂（如新增多模块页面、重构核心业务逻辑），无需跳过讨论直接开发。
- 操作步骤：输入指令 → 同步需求背景、待确认点 → 讨论达成共识后，再执行后续创建/开发指令。

**实战场景案例**：

- 场景：产品提出“新增用户个人中心页面”，但未明确页面包含的模块（如个人信息、订单列表、设置入口）、接口调用逻辑，且需要与后端确认数据返回格式。
- 操作：

```shell
# 输入
/opsx:explore 同步“新增我的任务页面”需求，与产品、后端讨论确定：页面包含5个模块 - 周日历模块默认展示当天；新客业绩列表调用新客业绩接口展示列表；老客经营列表调用老客经营接口展示列表；风险任务列表调用风险任务接口展示列表；日常任务列表调用日常任务接口展示列表。明确接口返回字段、页面交互逻辑。
# 输出：与 AI 讨论需求相关的功能模块设计。
# 讨论结束后，再使用 /opsx:new 或 /opsx:propose 执行后续操作。
```

***

### /opsx:new - 创建新变更

**使用方式**：

- 核心作用：创建新变更目录结构，为复杂需求的分步开发准备工作区，包含 proposal.md、design.md、specs/、tasks.md 等基础文件框架。
- 使用前提：需求已通过 explore 模式明确，需要分阶段生成文档并进行人工审核，或需求复杂度较高需要分步规划。
- 操作步骤：输入指令 → 输入变更名称 → 生成目录结构 → 使用 /opsx:continue 逐步生成文档。

**实战场景案例**：

- 场景：已通过 explore 模式明确"新增我的任务页面"需求，包含5个模块功能模块设计，需要分阶段生成 proposal、specs、design、tasks 并进行人工审核。
- 操作：

```shell
# 输入
/opsx:new
# 输出：提示输入变更名称，输入"add-my-tasks-page"后创建 changes/add-my-tasks-page/ 目录结构
# 后续使用 /opsx:continue 逐步生成各阶段文档
```

***

### /opsx:propose - 提案模式

**使用方式**：

- 核心作用：一步生成完整的变更规划，自动创建变更目录并生成 proposal.md、design.md、specs/、tasks.md 所有文档，无需分步操作。
- 使用前提：需求明确、复杂度中等，无需分阶段人工审核，希望一次性完成规划后直接进入开发阶段。
- 操作步骤：输入指令 + 需求描述 → 自动生成所有文档 → 审阅文档 → 使用 /opsx:apply 执行开发。

**实战场景案例**：

- 场景：需求明确，需要将客户页面筛选弹窗高度增加 80px，功能简单且边界清晰，无需分步讨论。
- 操作：

```shell
# 输入
/opsx:propose 兑换记录列表页面 标题改成：兑换记录页
# 输出：自动创建 changes/exchange-record-rename-title 目录，包含完整的 proposal.md、design.md、specs/、tasks.md
# 审阅文档后，使用 /opsx:apply 执行开发
```

***

### /opsx:continue - 继续变更

**使用方式**：

- 核心作用：在分步开发模式下，继续生成下一个阶段的文档，按顺序推进 proposal → specs → design → tasks 的生成流程，支持人工审核后再继续。
- 使用前提：已使用 /opsx:new 创建变更目录，且当前有未完成的文档生成阶段，需要人工审阅前一阶段文档后再继续。
- 操作步骤：审阅当前阶段文档 → 确认无误后输入指令 → 生成下一阶段文档 → 重复直至所有文档完成。

**实战场景案例**：

- 场景：使用 /opsx:new 创建了"add-my-tasks-page"变更，已生成 proposal.md 并审阅通过，需要继续生成 specs、design、tasks 文档。
- 操作：

```shell
# 输入
/opsx:continue
# 输出：生成下一阶段文档（如 specs/），审阅后再次输入 /opsx:continue 生成 design.md，依此类推直至 tasks.md 完成
# 所有文档完成后，使用 /opsx:apply 执行开发
```

***

### /opsx:apply - 应用变更

**使用方式**：

- 核心作用：根据 tasks.md 中的任务清单执行代码生成，可指定变更名称或在当前变更目录下直接执行，支持一次性执行所有任务或分步执行。
- 使用前提：变更目录已创建且 tasks.md 已生成并审阅通过，准备进入代码开发阶段。
- 操作步骤：确认 tasks.md 无误 → 输入指令（可指定变更名称） → AI 按任务清单生成代码 → 验证代码。

**实战场景案例**：

- 场景："exchange-record-rename-title"变更的 tasks.md 已准备完成，需要执行代码生成，将页面导航栏标题从"兑换记录"修改为"兑换记录页"。
- 操作：

```shell
# 在任意目录下，通过指定变更名称执行
/opsx:apply exchange-record-rename-title
# 或 先进入变更目录，然后直接执行（无需指定名称）
cd changes/exchange-record-rename-title
/opsx:apply
# 输出：根据 tasks.md 中的任务清单逐步生成代码
# 代码生成完成后，使用 /opsx:verify 进行验证
```

***

### /opsx:verify - 验证变更

**使用方式**：

- 核心作用：对已完成的变更进行全面验证，检查代码规范、代码与设计文档的一致性、字段显示与接口返回匹配度、接口对接是否正常等，生成验证报告。
- 使用前提：/opsx:apply 已执行完成，代码已生成，准备进行上线前的验证工作。
- 操作步骤：代码开发完成 → 输入指令 → 查看验证报告 → 根据报告修复问题 → 再次验证直至通过。

**实战场景案例**：

- 场景："exchange-record-rename-title"变更的代码已生成，需要验证页面导航栏标题是否正确修改，代码是否符合规范，字段显示是否正常。
- 操作：

```shell
# 输入
/opsx:verify
# 输出：生成验证报告，包含代码规范检查、代码与文档一致性检查、字段匹配检查等结果
# 如发现问题，修复后再次执行 /opsx:verify 直至验证通过
```

***

### /opsx:archive - 归档变更

**使用方式**：

- 核心作用：将已完成并上线的变更归档到 changes/archive/ 目录，保持 changes/ 目录整洁，便于后续查找和追溯历史变更。
- 使用前提：变更已通过 /opsx:verify 验证通过，功能已上线，无需再进行修改或开发。
- 操作步骤：确认功能上线 → 输入指令 → 变更被移动到 archive 目录 → 归档完成。

**实战场景案例**：

- 场景："exchange-record-rename-title"变更已上线，页面导航栏标题修改功能运行正常，需要归档该变更以保持工作区整洁。
- 操作：

```shell
# 输入
/opsx:archive
# 输出：将 changes/exchange-record-rename-title/ 目录移动到 changes/archive/ 目录
# 归档完成后，changes/ 目录下不再显示该变更
```

***

### /opsx:bulk-archive - 批量归档

**使用方式**：

- 核心作用：一次性批量归档多个已完成的变更到 changes/archive/ 目录，适用于版本发布后或定期清理工作区场景，避免逐个归档的繁琐操作。
- 使用前提：有多个已验证通过且上线的变更需要归档，changes/ 目录下存在多个待归档的变更目录。
- 操作步骤：确认所有待归档变更已上线 → 输入指令 → 选择或确认批量归档的变更列表 → 执行批量归档。

**实战场景案例**：

- 场景：v2.0 版本发布完成，changes/ 目录下有 "exchange-record-rename-title"、"update-exchange-record-background-color" 2 个已上线的变更需要批量归档。
- 操作：

```shell
# 输入
/opsx:bulk-archive
# 输出：显示待归档变更列表，确认后批量将所有已完成变更移动到 changes/archive/ 目录
# 批量归档完成后，changes/ 目录仅保留进行中的变更
```

***

### /opsx:ff - 快进模式

**使用方式**：

- 核心作用：跳过所有设计阶段，直接生成开发任务并执行，适用于紧急、简单的需求，如文案修改、UI微调等，无需完整的设计文档。
- 使用前提：需求极其简单明确，仅需修改文案、调整样式或修复小bug，无需讨论、无需设计文档，需要快速上线。
- 操作步骤：输入指令 + 简单需求描述 → 跳过设计直接生成 tasks.md → 立即执行代码生成 → 快速验证上线。

**实战场景案例**：

- 场景：兑换记录列表页面 标题改成：兑换记录页，需求简单明确，需要快速上线。
- 操作：

```shell
# 输入
/opsx:ff 兑换记录列表页面 标题改成：兑换记录页
# 输出：跳过 proposal、specs、design 阶段，直接生成 tasks.md 并执行代码生成
# 快速验证后即可上线
```

***

### /opsx:sync - 同步规范

**使用方式**：

- 核心作用：将变更目录 specs/ 中的功能规格合并到项目主规范库 specs/ 目录，实现规范沉淀和复用，便于后续开发参考。
- 使用前提：新功能已稳定上线运行一段时间，变更中的 specs/ 内容经过验证可作为通用规范，需要沉淀到主规范库。
- 操作步骤：确认功能稳定 → 输入指令 → 将变更 specs/ 内容合并到主 specs/ 目录 → 规范沉淀完成。

**实战场景案例**：

- 场景："exchange-record-rename-title"变更已上线运行 2 周，功能稳定，相关功能规格可作为通用规范沉淀到主规范库。
- 操作：

```shell
# 输入
/opsx:sync
# 输出：将 changes/exchange-record-rename-title/specs/ 目录下的规范合并到项目根目录的 specs/ 目录
# 规范同步完成后，后续开发可参考主规范库
```

***

### /opsx:onboard - 引导教程

**使用方式**：

- 核心作用：提供交互式引导教程，帮助新用户快速了解 OpenSpec 工具的功能、流程和最佳实践，通过 step-by-step 的方式熟悉各个指令的使用。
- 使用前提：首次使用 OpenSpec 工具、新成员入职需要熟悉工具流程、或需要复习工具使用方法。
- 操作步骤：输入指令 → 跟随交互式引导 → 了解工具功能和流程 → 完成引导教程后开始实际使用。

**实战场景案例**：

- 场景：新入职开发人员需要了解 OpenSpec 工具的使用方法，包括如何从需求探索到代码生成、验证、归档的完整流程。
- 操作：

```shell
# 输入
/opsx:onboard
# 输出：启动交互式引导教程，逐步介绍各个指令的功能和使用场景
# 跟随引导完成教程后，即可开始使用 OpenSpec 进行实际开发
```

## 三、目录结构

```shell
openspec/
├── changes/            # 变更目录
│   ├── archive/        # 已归档变更目录
│   └── <change-name>/
|       ├── .openspec.yaml # 变更配置文件，包含schema，created
│       ├── proposal.md # 需求说明（what & why）：说明为什么要做这个变更
│       ├── design.md   # 技术设计（how）：说明如何实现这个变更
│       ├── specs/      # （可选）功能规格目录，重大需求变更涉及config.yaml文件改动时用上
│       │   └── <spec-name>/
│       │       └── spec.md # 功能规格：定义系统应该做什么
│       └── tasks.md # 实现步骤：详细的开发步骤清单
├── specs/          # 主规范库（sync后生成）
└── config.yaml    # OpenSpec 配置
```
